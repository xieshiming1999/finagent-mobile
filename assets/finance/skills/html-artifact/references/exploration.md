# Exploration & Planning Patterns

## When to use

The user doesn't know what they want yet — they want to **see options**, not
read a recommendation. Or they've decided and need a **handoff document** that
makes the plan scannable at a glance.

## Templates

### 01 — Three Code Approaches

**Shape:** 3 columns (or stacked cards), each with a title, description,
trade-off callout, and a short code snippet.

**Key elements:**
- Header with problem statement
- 3 approach cards with: name, description, pros/cons badges, code sample
- A "Recommended" ribbon on one card
- Summary table at bottom

**Adapt for:** any "which way should we..." question — architecture choices,
vendor comparisons, schema update strategies.

### 02 — Visual Design Directions

**Shape:** 2×2 grid of rendered mockups, each with a small label and
one-sentence rationale.

**Key elements:**
- 4 live-rendered variations (colors, layouts, type choices)
- Each has a name and tone descriptor
- Footer asks "which direction resonates?"

**Adapt for:** brand explorations, landing page comps, component aesthetics.

### 16 — Implementation Plan

**Shape:** Left-rail timeline + right-side detail panels.

**Key elements:**
- Milestone timeline with dates and status badges
- Data-flow or architecture diagram (inline SVG)
- Inline mockup (miniature UI render) of the end state
- Risky code block highlighted
- Risk table (probability × impact matrix)

**Adapt for:** project plans, sprint planning, feature proposals, RFC visuals.
